
# 产品需求文档 (PRD)：多模态浏览器 Agent (Space-Aided Agent)

## 1. 文档概述
本文档旨在定义一个基于视觉引导与双路 RAG（检索增强生成）机制的浏览器 Agent 系统。该系统通过视觉子智能体提供空间与语义线索，结合高效的 DOM 清洗与双表索引技术，实现在超长 HTML 环境下的精准定位与低 Token 消耗的操作决策。

## 2. 系统架构
系统采用“预处理 - 感知 - 索引检索 - 融合 - 决策”的五阶段流水线。

### 2.1 核心组件
- **HTML Cleaner**: 静态 HTML 净化器。
- **Visual Subagent**: 视觉感知子智能体（Vision-based）。
- **Indexer**: 双表索引构建器（Space Table & Key Table）。
- **Dual-RAG Engine**: 空间与语义双路检索引擎。
- **Fusion Module**: 全局去重与 DOM 重组模块。
- **Main LLM**: 主决策大模型。
- **Agent Tools (Middleware)**: 动作解析与执行中间层。
- **Agent Memory**: 具备总结压缩机制的记忆系统。

## 3. 详细功能需求

### 3.1 HTML 深度预处理 (Preprocessing)
**需求描述**：在进入任何索引逻辑前，必须对原始 DOM 进行清洗，以移除 70% 以上的无效噪声。

**处理规则**：
1. **彻底移除块**: `script`, `style`, `noscript`, `iframe` 及其内部内容。
2. **过滤无关标签**: `link`, `meta`, `base`。
3. **图形简化**: `svg` 内部路径全部替换为 `...`，仅保留标签。
4. **资源占位**: 将所有 `src` 和 `srcset` 中的 Base64 Data URIs 替换为 `...`。
5. **标签白名单**: 仅保留交互及语义相关的标签（如 `div`, `span`, `a`, `button`, `input`, `form`, `ul`, `li`, `h1-h6`, `table` 等）。
6. **文本截断**: 标签间的文本长度若超过 **30 字符**，则保留前 30 个并加 `...`。
7. **格式优化**: 压缩多余空白符，并在标签间插入换行符以增强层级感。

### 3.2 双表索引构建 (Indexing)
**需求描述**：构建以 `node_id` 为核心的两张查询表，实现 $O(1)$ 或 $O(\log N)$ 的检索效率。

- **Key Table (关键词表)**:
    - **结构**: `Map<String, List<NodeID>>`
    - **内容**: 索引节点的 `innerText`、`placeholder`、`aria-label` 等关键文本属性。
- **Space Table (空间坐标表)**:
    - **结构**: `List<{id: NodeID, center: [x, y], rect: Rect}>`
    - **内容**: 存储所有清洗后可见元素的屏幕中心坐标及边界框。

### 3.3 视觉预感知 (Visual Subagent)
**需求描述**：通过截图分析，为后续 RAG 提供引导参数。
- **输出参数**:
    - `focus_point`: 目标区域中心坐标 $(x, y)$。
    - `keywords`: 目标语义关键词。
    - `status`: 当前已完成任务的总结。
    - `issues`: 识别出的阻碍（如弹窗覆盖、加载中等）。
    - `is_completed`: 视觉确认任务是否已完成。

### 3.4 双路 RAG 检索 (Dual-RAG)
系统并行启动以下两路检索逻辑：

1. **Key-RAG (语义路)**:
    - 输入 `keywords`，在 Key Table 中匹配最相关的 $top\text{-}k$ 个核心节点。
2. **Space-RAG (空间路)**:
    - 输入 `focus_point`，采用 **视觉聚焦半径 (Visual Focus Radius)** 算法。
    - 检索所有中心点位于以 `focus_point` 为中心、边长为页面长宽 $p\%$ 的矩形区域内的节点。

**路径扩展规则 (Graph Distance Expansion)**：
对于命中的核心节点，沿 DOM 树向上/向下寻找路径距离 $n$ 以内的所有节点。此举旨在保留 UI 组件的完整性（如 Label 与其对应的 Input）。

### 3.5 融合与去重 (Fusion & Deduplication)
**需求描述**：消除两路检索产生的数据冗余。
- **逻辑**:
    1. 获取两路 RAG 产生的节点合集。
    2. 基于 `node_id` 进行全局去重。
    3. **顺序重组**: 按照原始 HTML 中的出现顺序排列节点，确保生成的局部 HTML 符合规范结构。
    4. 构建最终的局部 HTML 代码块。

### 3.6 动作执行支持 (Action Execution)
**需求描述**：支持复杂且鲁棒的浏览器交互。
- **多动作支持**: 主模型可单次返回多个动作（分号分隔），如 `fill(id=1, value='abc'); click(id=2)`。
- **候选集支持 (Fallbacks)**: 每个动作可携带备选方案（`||` 分隔），如 `fill(id=1) || click(id=1); keyboard.type('abc')`。
- **合法性验证**: 执行前验证元素是否可见、是否支持对应操作（如 Div 不支持 Fill），失败时自动尝试 Fallback。
- **多标签页支持**: 支持 `switch_tab(index)`、`close_tab()` 及感知所有打开的标签页。

### 3.7 记忆系统 (Memory System)
**需求描述**：管理长任务下的执行上下文。
- **记录内容**: 每步的思考、实际执行的动作列表、各动作执行结果、视觉感知状态。
- **总结压缩**: 当历史记录超过阈值时，利用 LLM 将旧记忆压缩为“前期任务总结”，保留最近几步的详细记录。

## 4. 非功能需求与技术参数

| 参数项 | 设定值 | 说明 |
| :--- | :--- | :--- |
| **$top\text{-}k$** | 3 - 5 | Key-RAG 提取的核心节点数量 |
| **$n$ (Path Depth)** | 2 | 节点扩展的 DOM 路径层级深度 |
| **$p\%$ (Focus Radius)**| 5% - 10% | 空间检索的视野覆盖百分比 |
| **Text Threshold** | 30 | 节点纯文本截断阈值 |
| **Deduplication ID**| `data-agent-id` | 用于跨表追踪的唯一 ID |
| **Execution Policy**| Sequential | 动作列表按序执行，遇失败记录状态 |

---

## 5. 业务流程 (Workflow)
1. **感知**: 视觉 Subagent 截图并输出坐标、关键词及任务状态。
2. **构建**: 同步运行 `clean_html` 并构建双路索引。
3. **检索**:
    - 基于半径 $p\%$ 检索空间节点。
    - 基于关键词检索语义节点。
4. **融合**: 路径扩展、全局去重并按原始顺序重组。
5. **决策**: 主 LLM 结合局部 HTML 与压缩后的记忆，输出动作指令。
6. **执行**: 工具层解析动作列表与候选集，反馈执行结果至下一轮记忆。

---

## 6. 异常处理
- **RAG 为空**: 若双路 RAG 均未命中有效节点，主 LLM 应根据视觉描述尝试全局操作（如 `scroll` 或 `goto`）。
- **执行失败**: 若所有候选动作均失败，记录错误并由主 LLM 在下一轮尝试修复方案。
- **死循环检测**: 记忆系统应辅助检测是否在重复相同的失败动作。
